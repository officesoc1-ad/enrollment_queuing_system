Next.js Enrollment Queuing System
Project Documentation

This document outlines the User Flow, Setup Guide, and System Architecture for the Next.js Enrollment Queuing System, emphasizing its high-performance, cost-optimized hybrid architecture.

1. User Flow

1.1 Student Registration & Tracking

1. Registration (/register): Students visit the portal, provide their ID, Course, and Year Level, and are placed into the corresponding enrollment queue block. Form validation is strictly handled by Zod.

2. Queue Ticket Issued: Upon successful entry creation into Supabase, the student is given a unique Queue Number and redirected to their personal tracking page.

3. Interactive Tracking (/student/[id] or /track): Students monitor their live position, wait time, and current serving number. To drastically reduce server costs, the tracking UI relies on "visibility-aware adaptive polling" (every 60 seconds) rather than WebSockets, pausing entirely if the browser tab is hidden.

1.2 Admin Management

1. Dashboard Core (/admin/dashboard): Authenticated staff access a comprehensive dashboard separated into "Queues", "Schedules", "Courses", and "Admins" administration.

2. Queue Operations: Admins can trigger the "Call Next" function, moving students to "serving". They can also mark students as "completed" or "skipped" (No-show).

3. Optimistic Updates & Sync: Actions use Optimistic UI for instant feedback, while in the background, lightweight direct Supabase Realtime listeners debounce and reconcile the exact system state without taxing Vercel serverless functions.

1.3 Live Queue Board

1. Public Display (/queue): A multi-column public board displays active queues broken down by Course and Year Level, showing the "Now Serving" tickets.

2. WebSocket Distribution: Unlike the student pages, the centralized live board uses Supabase Realtime Postgres Changes to listen to state modifications, keeping visual changes instantaneous without page reloads.

2. Setup Guide

To set up the Next.js application locally, ensure you meet the prerequisites.

Prerequisites

- Node.js (v18.0 or newer)

- A Supabase Project (with enabled Realtime on public schema)

- Git

Environment Configuration

Create a `.env.local` file at the root of the project with the following keys:

NEXT_PUBLIC_SUPABASE_URL=your-project-url.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key

SUPABASE_SERVICE_ROLE_KEY = your_supabase_service_role_key

Installation & Execution

1. Install dependencies via npm:

npm install

2. Run the Next.js development server:

npm run dev

3. Ensure Supabase migrations/SQL scripts have been executed in your remote Supabase SQL Editor to construct tables (queue_configs, queue_entries, courses, schedules) and enable Replica Identity/Realtime.

3. System Architecture & Optimizations

This system was heavily refactored to implement a "Hybrid Architecture" directly designed to mitigate Vercel Serverless Function invocations during high-traffic 6-day enrollment events.

3.1 Core Technology Stack

- Framework: Next.js 16 (App Router)

- UI Layer: React 19

- Database & Auth: Supabase (PostgreSQL)

- Data Validation: Zod

3.2 Data Strategy Framework

1. Direct Supabase Reads (Bypass Next.js API): To stay within Vercel Free Plan limits, read-heavy operations (Live Board updates, Admin Dashboard refresh, Student status polling) invoke the direct Supabase REST/WebSocket client (lib/supabase-client.js) instead of hitting /api routes. This drives Vercel function consumption effectively to zero for data consumption.

2. Next.js API Write Handling: Data modifications requiring elevated privileges (like an admin calling the next student or saving new schedules) use standard authenticated Next.js API routes.

3. Adaptive Polling: Instead of maintaining 500+ unutilized WebSocket pipelines for individual students checking their status, the client relies on smart 60-second interval polling connected to the Page Visibility API. Polling halts when the tab is inactive.

3.3 Real-Time Strategy

Supabase Realtime is utilized judiciously on centralized single-displays (Admin Dashboard and Live Display) where state sync is critical. Updates to `queue_entries` and `queue_configs` trigger event listeners, which use a debounced direct-fetch command to retrieve true state differences across the network.
